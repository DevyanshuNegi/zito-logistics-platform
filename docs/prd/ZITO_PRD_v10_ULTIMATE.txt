1. Product Vision
ZITO is a unified logistics super platform built for Africa-scale operations. It consolidates fragmented transport, warehousing, and last-mile delivery services into a single intelligent platform, enabling operators, transporters, drivers, warehouses, and customers to collaborate in real time.
Core Services
FTL (Full Truck Load) — long-haul freight transport
PTL / LCL (Partial Load) — consolidated cargo shipments
Courier / Last-mile delivery — direct-to-door delivery
Warehouse & storage services — managed storage with full inventory control
Containerized trucking — explicit 20ft and 40ft container movement
Rail / SGR corridor logistics — container rail movement such as Mombasa port to inland rail-linked nodes
Fleet & transporter management — vehicle lifecycle and driver coordination
Multi-country logistics — Africa-ready, offline-capable operations

1.A PLATFORM POSITIONING (Critical for Launch - May 13 Update)

ZITO is a LOGISTICS PLATFORM, not a logistics operator:
✓ ZITO = Infrastructure that connects supply + demand
✓ ZITO owns NO trucks, NO warehouses, NO drivers
✓ ZITO enables transporters, drivers, warehouses, couriers to work together efficiently
✓ ZITO takes commission on coordination value created (not freight value)

What ZITO is NOT:
✗ NOT competing with DHL/FedEx/Aramex (they could be partners)
✗ NOT a delivery company
✗ NOT an operator
✗ NOT copying competitor business models

Business Model (90% B2B):
- 90% Revenue from B2B clients: Transporters, Courier Companies, Warehouses, Corporate Shippers
- 10% Revenue from B2C: Individual/small senders using platform
- Revenue streams: Platform commissions (5-10% per booking) + B2B subscriptions + services
- Profit margin: 70-90% (software economics, not logistics operator model)
- Unit economics: Scale without capital (no trucks/warehouses to buy)

Platform Economics:
- Transporters benefit: Better load matching, reduced downtime, automated dispatch
- Drivers benefit: Jobs pushed via app, earnings tracking, daily payouts
- Shippers benefit: One platform for all services (FTL, PTL, Courier, Warehouse)
- ZITO benefits: Commission on every transaction, growing network = growing value

Network Effects (Competitive Moat):
- More transporters → Better coverage → Faster matching → More shippers attracted
- More shippers → More demand → More transporters attracted
- Result: Winner-take-most market dynamics (first to scale dominates)

Canonical flow rule:
OTP step usability rule:
- the OTP screen must focus the code-entry field immediately on entry
- the OTP screen should auto-submit once a complete valid six-digit code is captured by paste, platform autofill, or compatible browser-assisted OTP capture unless the active compliance flow explicitly requires a separate final confirmation action
contact step -> OTP step for all public users -> password step only for email users -> role-based redirect after successful verification.
Phone login completes after OTP only.
Email login requires password after OTP verification and must not expose password entry on the initial contact step.
When email OTP trouble occurs and the account supports passwords, the same login screen must unlock email/password fallback without sending the user to a separate login page.
OTP Handling & Security Rules:
Target defaults:
OTP validity = 5 minutes
Resend cooldown = 30 seconds
UI must show resend countdown before another code can be requested
Maximum resend attempts = 4 resends after the first code, 5 total OTP requests
OTP validity: configurable (e.g., 2–5 minutes)
Resend OTP allowed after cooldown (e.g., 30 seconds)
Maximum resend attempts limit (e.g., 3–5 attempts)
Maximum verification attempts limit (e.g., 5 tries)
Rules:
Expired OTP must be rejected
Multiple failed attempts → temporary account lock
OTP must be single-use only
OTP must be stored hashed at rest (using bcrypt with 10 rounds)
OTP verification must use secure comparison (bcrypt.compare) to prevent timing attacks
Verification target must be masked in the UI
Failed OTP responses must show attempts remaining whenever the account is not yet locked
OTP provider failures must be distinguished from invalid credentials; provider-delivery failure, timeout, or upstream outage must surface as an integration / service-availability error, not as a false password or user-not-found response
Auth health monitoring must include OTP provider connectivity for SMS and email channels used in the active market
OTP entry surfaces must accept full-code paste and system autofill, not only one-digit manual entry
Phone-auth OTP UX should auto-fill wherever the platform supports it: iPhone through one-time-code autofill, Android through SMS-OTP or app-hash-compatible delivery, and compatible browsers through origin-bound WebOTP patterns where the SMS provider template supports it

Delivery OTP Rate Limiting:
- Applies to all delivery confirmation OTP attempts across the system
- Maximum 5 consecutive failed verification attempts per booking
- After 5 failed attempts, the booking enters a locked state for 15 minutes
- During lockout, any additional OTP verification attempts are rejected with lockout message
- Lockout is automatically cleared after 15 minutes
- Successful OTP verification resets the attempt counter to 0 and clears any existing lockout
- Rate limiting applies across all delivery verification paths: driver app status update and warehouse scan-based delivery confirmation
- Failed attempt counter increments regardless of verification method (driver status update vs warehouse scan)

4.B AUTHENTICATION UPDATES: REGISTRATION VERIFICATION
Registration must not immediately activate an account. Registration creates a pending account and starts the verification workflow.
Mobile OTP verification during registration:
- every registration must capture a valid mobile number with editable country code and local phone number
- the system must send a one-time password to the registered mobile number
- the user must verify the mobile OTP before the account can proceed to KYC submission

Email OTP / email verification during registration:
- where an email is supplied, the system must verify ownership through email OTP or equivalent email-verification link
- email verification must be completed before final account activation
- if a user has no email where policy permits phone-only registration, account activation must still require mobile verification and KYC approval
OTP resend and cooldown rules:
- OTP resend must enforce cooldown to prevent repeated requests
- OTP records must expire after the configured validity window
- repeated failed verification attempts must lock the OTP target temporarily
- resend count, attempts remaining, expiry time, and lock state must be represented in API responses where safe for the user experience
- OTP errors must be context-aware; phone-flow errors must not show email/password errors, and email-flow errors must not leak stale phone errors
Registration verification workflow:
- registration state must be isolated from login state
- changing phone or email during registration must clear old OTP state for that target

4.C KYC & VERIFICATION SYSTEM
KYC workflow for all user types:
- all user types must support a KYC profile with required documents derived from role, country, service scope, and compliance policy
- KYC upload must happen after contact verification and before admin approval
- users must be able to review their submitted KYC packet before final submission
- KYC submission must move the account to pending review or under review; upload alone cannot activate any account
Document upload system:
- document upload must support required document type, issuing country, document/reference number where applicable, issue date, expiry date where applicable, and file evidence
- KYC upload must be camera-first for identity and verification evidence; the primary user action must open live camera capture rather than a generic file picker
- document evidence must support image/PDF only where policy allows; live-camera-only evidence must not allow gallery/library upload
- sensitive KYC images such as National ID, Passport, driving license, driver selfie/photo, authorized signatory ID, and business-document evidence required for verification must include capture-source metadata
- where a KYC requirement is marked live-camera-only, the backend must reject uploads that do not declare an approved live-camera capture source and capture timestamp
- uploaded files must be linked to user, role, document type, upload timestamp, and review state
- documents must be stored in secure private storage in production, with signed/authorized access for review
Verification statuses:
- NOT_STARTED: user has not begun KYC
- CONTACT_VERIFICATION_PENDING: mobile/email verification incomplete
- KYC_PENDING: KYC documents not fully submitted
- PENDING_REVIEW: KYC packet submitted and awaiting admin review
- UNDER_REVIEW: admin/compliance review in progress
- RESUBMISSION_REQUIRED: one or more required documents rejected or missing
- VERIFIED: required KYC has been approved
- ACTIVE: account activated after verification and approval
- REJECTED: account or KYC packet rejected according to policy
- SUSPENDED: account disabled due to expiry, compliance, fraud, admin action, or policy
Approval and rejection workflows:
- admin or super admin must review documents at document level first
- approval must capture reviewer identity, timestamp, document status, and notes where applicable
- rejection must capture reviewer identity, timestamp, mandatory rejection reason, and resubmission instructions
- account verification status must be recalculated from the reviewed document set
- activation must remain separate from verification and must require authorized internal action or approved activation automation
Resubmission logic:
- rejected or expired documents must be resubmittable without deleting the audit trail of the previous submission
- resubmission must create a new document version or review record linked to the original requirement
- users must see which document failed, why it failed, and what must be uploaded again
- resubmission must return the affected requirement to pending review or under review
Admin verification controls:
- admin and super admin must have a Verification Control desk for customer KYC, driver KYC, partner KYC, fleet verification, document review, rejection handling, expiry monitoring, and resubmission tracking
- the desk must allow filtering by role, status, document type, overdue review, expiry risk, rejected items, and resubmission-required items
- the desk must expose approve, reject, request resubmission, suspend, activate, and audit-history actions according to RBAC
- every verification decision must be written to immutable audit logs

4.G ENTERPRISE SECURITY ARCHITECTURE & PRODUCTION READINESS
Security architecture baseline:
- ZITO production must follow a defense-in-depth model aligned to OWASP ASVS, OWASP Top 10 access-control guidance, and secure file-upload controls.
- The backend is the authoritative trust boundary for authentication, authorization, payments, KYC, fleet verification, dispatch, tracking, and account activation.
- Frontend and mobile validation are usability aids only; every critical rule must be enforced server-side.
- Every API that reads or mutates a business object must enforce both role authorization and object-level ownership or scope checks.
- Sensitive workflows must be auditable: login, OTP, password reset, KYC upload, KYC review, fleet verification, payment initiation, payment confirmation, refund, reversal, wallet mutation, dispatch acceptance/rejection, live tracking access, and admin account changes.

Authentication and session requirements:
- Production JWT secrets must be provided only through managed secret storage; fallback/default JWT secrets are forbidden in deployed environments.
- Access tokens must be short-lived and refresh tokens must be rotated, revocable, and stored server-side or in secure HttpOnly cookies where applicable.
- Browser storage of long-lived refresh tokens is not production-approved because XSS can expose tokens.
- Session validation must reject unknown session IDs in production; validation must not silently create a session record from a presented JWT.
- OTP send, resend, verification, password reset, and login attempts must have contact-level and IP/device-level throttling.
- Admin, refund, payout, role-change, force-logout, KYC approval, and account activation actions must require re-authentication or step-up approval.

API and RBAC requirements:
- RBAC must be enforced on every protected controller, and role checks must be paired with object-scope checks for customer, driver, transporter, courier company, warehouse partner, corporate, agency staff, admin, and super admin users.
- Public endpoints must be explicitly documented and must not expose sensitive object details.
- Payment, booking, wallet, featured-listing, subscription, dispatch, and document-review mutations must require server-enforced idempotency.
- Errors returned to clients must be context-aware and must not expose stack traces, provider secrets, SQL details, internal IDs beyond the requested object, or cross-flow auth state.
- Swagger/API docs must be disabled or access-controlled in staging and production.

KYC, document, and vehicle evidence security:
- Production KYC and vehicle evidence must be stored in private object storage with encryption at rest, signed short-lived read URLs, and no unauthenticated public paths.
- Uploaded files must be validated by declared MIME type, detected file signature, extension allowlist, size limit, and malware/AV scanning before review.
- Live-camera-only evidence must include capture source, capture timestamp, user ID, document type, and submission context in structured metadata.
- KYC images, driver live photos, vehicle photos, insurance evidence, permit evidence, POD images, invoices, and waybills must not be served from public static local upload directories in production.
- Rejected or resubmitted documents must preserve immutable audit history while preventing ordinary users from viewing other users' documents.

Payment and financial security:
- Payment initiation must verify the authenticated user is allowed to pay the booking or invoice target.
- Payment lookup, retry, refund, reversal, escrow, wallet, disbursement, featured-listing, and subscription operations must enforce owner/admin scope.
- Provider callbacks must be authenticated or verified with provider signatures, source allowlists, shared secrets, or equivalent M-Pesa control before state mutation.
- Wallet and ledger updates must be transactional, idempotent, immutable, and reconciled against provider references.
- Duplicate payment, invoice overpayment, escrow bypass, refund abuse, and replay attacks must be covered by automated tests before go-live.

Real-time tracking security:
- WebSocket connections must authenticate the user before joining rooms.
- Drivers may only publish GPS for their own driver profile and assigned active bookings.
- Customers may only subscribe to tracking rooms for bookings they own or are authorized to view.
- Admin live map rooms must require admin/super-admin authorization.
- GPS updates must be rate-limited and checked for impossible movement, spoofing indicators, and booking-state validity.

Infrastructure and deployment hardening:
- Production must require HTTPS/TLS, secure CORS origin allowlists, managed secrets, least-privilege database credentials, private storage permissions, backup and restore tests, centralized logs, alerting, and monitored health checks.
- CI/CD must include dependency audit, secret scanning, type/build checks, unit tests, integration tests, migration checks, and security regression tests.
- Production deployments must not run with development OTP exposure, test redirects, simulated payment providers, public debug logs, or local-only provider fallbacks.
- Database access must use least privilege, TLS, encrypted backups, migration review, and point-in-time recovery where supported.
- Rate limiting, WAF/DDoS protection, request-size limits, and upload-size limits are mandatory for public production.

Audit logging and monitoring:
- Audit logs must capture actor ID, target ID, action, timestamp, IP address, device/user-agent, request correlation ID, result, and approval reason where applicable.
- Security monitoring must alert on brute force, OTP abuse, repeated payment failures, refund spikes, role changes, account activation spikes, suspicious login, high-volume uploads, GPS anomalies, and unauthorized access attempts.
- Audit logs must be tamper-resistant and retained according to compliance and finance requirements.

Production readiness checklist:
- No critical or high unresolved security findings.
- All sensitive uploads private, encrypted, signed, validated, and malware-scanned.
- JWT/session/refresh-token storage and revocation production-approved.
- RBAC and object ownership tests pass for all user types.
- Payment idempotency, callback verification, wallet integrity, and reconciliation tests pass.
- WebSocket authentication and tracking authorization tests pass.
- Rate limiting and abuse controls enabled for auth, OTP, uploads, payments, and public APIs.
- Production secrets, CORS, HTTPS, backups, monitoring, and incident response runbooks verified.
- PRD, API docs, test cases, and deployment runbook updated before module lock.

4.I OTP PROVIDER ABSTRACTION AND TEST OTP MODE
OTP provider architecture:
- OTP delivery must be abstracted behind a provider layer so ZITO can switch between test, Twilio Verify, and future providers without changing auth controllers, frontend OTP screens, or user-facing auth flow.
- Twilio Verify remains the production-ready SMS OTP provider and must not be removed.
- The provider switch must be environment-driven through OTP_MODE.
- Supported OTP modes:
  - OTP_MODE=test: internal QA mode, no real SMS sent, configured TEST_OTP_CODE is accepted.
  - OTP_MODE=twilio: Twilio Verify SMS provider.
  - OTP_MODE=firebase: future-ready placeholder; must fail clearly until implemented.
  - unset/local fallback: local development fallback for non-production developer flows.
Test OTP mode policy:
- TEST OTP MODE exists only to avoid SMS cost during internal QA and local development.
- TEST_OTP_CODE must be configurable and must default only in non-production.
- TEST OTP MODE must preserve OTP expiry, resend cooldown, resend limits, failed-attempt tracking, and lockout rules.
- TEST OTP MODE must not change frontend UX, resend countdown behavior, loading states, or API contracts.
Production OTP restrictions:
- NODE_ENV=production with OTP_MODE=test is forbidden and must block application startup.
- NODE_ENV=production with OTP_MODE=firebase is forbidden until the provider is implemented and approved.
- production must use Twilio Verify or another approved live OTP provider with real delivery and provider health monitoring.
Provider failover strategy:
- Provider failures must be surfaced as OTP delivery/provider errors, not as invalid credentials.
- Failover to another live provider may be added only after audit, rate-limit, logging, and compliance controls match Twilio mode.
- No production failover may silently fall back to test OTP mode.
OTP environment handling and logging:
- OTP logs must mask destination values and must not log plaintext OTPs in production.
- Test mode may expose TEST_OTP_CODE only through controlled internal QA documentation, never in production responses.
- OTP verification and delivery failures must be audit/security observable without leaking sensitive code values.
Authentication stabilization and session lifecycle:
- The OTP login lifecycle is: phone/email input -> provider-selected OTP dispatch -> latest OTP record created with TTL/cooldown metadata -> temp OTP JWT issued -> OTP verification against the same normalized contact -> optional email-password completion -> session record creation -> access JWT generation -> frontend session persistence -> status-aware redirect.
- In OTP_MODE=test, configured TEST_OTP_CODE, defaulting to 123456 only outside production, must verify successfully for phone and email login without Twilio, SMS cost, or provider fallback conflicts.
- Test OTP mode must still create and expire OTP records, invalidate older resend records, enforce resend cooldown, enforce maximum resend count, track failed attempts, and lock repeated failures.
- Twilio must never be invoked when OTP_MODE=test. Twilio may be enabled only when OTP_MODE=twilio or when explicitly configured as an approved live provider outside local test mode.
- Non-suspended users may authenticate during onboarding. PENDING, VERIFIED, and REJECTED users receive a limited session that can access profile and verification routes only; ACTIVE users receive normal module access.
- SUSPENDED users must be blocked before OTP dispatch and before session creation.
- Every access JWT must include a server-backed session id and user status. JWT validation must reject missing/invalidated/expired sessions and must not create dashboard access for incomplete users.
- Frontend OTP state must be flow-isolated. Switching phone/email, changing the OTP target, retrying login, or resending OTP must clear stale errors and use only the latest temp token/contact pair.
- Auth debugging logs in development may show provider mode and masked destination only. Production logs must never expose OTP values, auth tokens, refresh tokens, or full phone/email identifiers.
- Pending-review login logic must redirect incomplete users to Complete Verification, not dashboards. Pending Approval is shown only after required KYC documents are uploaded/submitted.
Corrected test OTP flow and stale-state prevention:
- Starting a fresh phone or email OTP request must clear any previous OTP session, previous OTP input, previous verification error, previous password-gate state, auto-submit marker, cooldown state that belongs to another contact, and stale local auth/session data for the same flow.
- The OTP screen may open only after /auth/login returns a valid temp_token, contact, expiry, and resend metadata. The frontend must not call /auth/verify-otp before the user submits a 6-digit code or a valid browser WebOTP credential is received for the same contact/session.
- TEST OTP MODE must never depend on a provider-generated external SID/status. The backend must still create a local OTP row for expiry/cooldown tracking, but verification must accept TEST_OTP_CODE through the test provider when OTP_MODE=test.
- When a resend occurs, the previous active OTP row for that contact must be expired and only the newest temp token/contact pair may be used by the frontend.
- Frontend error messages must be context-aware: OTP send failure is a delivery/request problem, while OTP verification failure is a code/session problem. A send failure must not be shown as "OTP failed" before the OTP screen opens.
- Development auth debugging may log state transitions such as OTP_REQUEST_STARTED, OTP_REQUEST_ACCEPTED, OTP_VERIFY_STARTED, OTP_VERIFY_ACCEPTED, SESSION_CREATED, and AUTH_REDIRECT_SELECTED with masked contact and no token/code values.

4.K CURRENT-LOCATION-FIRST PICKUP EXPERIENCE
Pickup location selection must behave like modern mobility and delivery applications such as Uber, Bolt, Glovo, and Google Maps while remaining suitable for logistics operations.
Current-location-first rule:
- When a booking pickup page opens, the frontend should request browser/mobile location permission where available.
- If permission is granted, the system must detect GPS coordinates, estimate accuracy, reverse geocode the nearest readable area/address, and place "Use Current Location" at the top of pickup suggestions.
- The current-location suggestion must include the resolved address or nearest area, for example: "Use Current Location - Westlands, Nairobi, Kenya".
- Selecting current location must auto-fill pickup details, center the map on the coordinates, drop the pickup pin, and allow the user to adjust the pin manually before submitting.
Pickup suggestion priority:
- Pickup suggestions must be ordered as: 1. Current GPS location, 2. Nearby locations, 3. Recent pickup locations, 4. Popular destinations, 5. Manual search results.
- Manual search must continue to work even when GPS is unavailable or denied.
Permission and fallback handling:
- While GPS is loading, the UI must show a mobile-friendly loading state or shimmer for the current-location suggestion.
- If permission is denied, unavailable, inaccurate, or times out, the UI must show a graceful message and continue autocomplete/manual entry without blocking booking.
- Location errors must not break pricing, booking submission, RBAC, or backend APIs.
Reverse geocoding policy:
- Reverse geocoding may use browser coordinates with existing frontend map/geocoding utilities where available.
- If an exact address cannot be resolved, the UI must fall back to a readable "Current location" label with coordinates retained internally.
Mobile-first booking UX:
- The current-location quick action must be visually prominent, touch-friendly, and accessible.
- GPS detection should be fast, cancellable by user navigation, and must not repeatedly nag for permission after denial in the same page session.
- No sensitive location data may be logged in plaintext outside controlled development diagnostics.

4.L MODERN SUPER-APP BOOKING EXPERIENCE
ZITO booking UX must feel like a modern African logistics super app, inspired by Uber, Bolt, Glovo, Uber Eats, Careem, and premium mobility products while preserving enterprise logistics rules.
Booking experience principles:
- Booking must be map-first, search-first, and mobile-first. The first visual focus must be a live map surface with a bottom-sheet interaction layer and a large "Where to?" search action, not an admin-style form.
- Desktop modal/dialog booking search is not acceptable for the customer route flow. Location search must use a mobile bottom sheet that feels app-native on phones and remains bottom-anchored on desktop.
- The route lifecycle is: auto-detect pickup -> show current location -> ask "Where to?" -> search/select destination -> preview route -> choose service/vehicle -> show price/rate state -> add contacts/details -> confirm booking.
- Pickup and drop-off selection must use a bottom-sheet/modal pattern with large touch-friendly rows, current location first, saved/recent/popular locations, and manual search as the fallback.
- Location search should feel smooth and fast, with clear loading states, graceful permission denial, and no blocking if GPS or reverse geocoding fails.
Map-first architecture:

4.J UNIVERSAL ONBOARDING AND ROLE-BASED KYC ARCHITECTURE
Universal onboarding rule:
- Every user role must pass through the same verification lifecycle before protected dashboard/module access is granted.
- The universal lifecycle is: Registration -> Mobile OTP verification -> Email verification where email is supplied or required -> Login -> Role detection -> Role-based KYC wizard -> Document upload -> Submit for review -> Under review -> Approval/resubmission/rejection -> Dashboard access after activation.
- KYC onboarding, document upload, approval workflow, account restrictions, and admin review must apply to individual customers, corporate customers, drivers, transporters, courier companies, warehouse partners, agents/brokers, agency users, and any future externally onboarded role.
- Pending approval is not the first post-login verification screen. Users with incomplete KYC must be routed to Complete Verification first.
Universal login rule:
- After login, the platform must detect role and verification state before redirecting to any workspace.
- If role-required KYC is incomplete, the user must be redirected to Complete Verification.
- If all required KYC is uploaded and submitted but not approved, the user must be redirected to Pending Approval / Under Review.
- If KYC is rejected or resubmission is required, the user must be redirected to Complete Verification with document-level rejection reasons and resubmission instructions.
- Only active and approved users may access protected role workspaces.
Role-based KYC document requirements:
- Individual customer: National ID or Passport; selfie/profile photo.
- Corporate customer: company registration; tax/VAT/KRA certificate; authorized person ID; company details and billing/contact details.
- Driver: National ID; driving license; PSV/transport permits where applicable; live-camera-only profile photo.
- Transporter: company registration; tax/VAT/KRA certificate; vehicle documents; insurance; driver assignment details.
- Courier company: company registration; operating license; fleet details; insurance where applicable.
- Warehouse partner: warehouse ownership or lease proof; compliance certificates; company registration; operating permits where applicable.
- Agent / broker: National ID; business proof; address/contact verification.
Dynamic KYC document engine:
- The backend must expose role-based KYC requirements derived from role, country, business type, service scope, and compliance policy.
- The frontend KYC wizard must load requirements dynamically instead of hardcoding a corporate-only checklist.
- The engine must validate missing documents, rejected documents, expired documents, resubmission-required documents, and submitted documents per role.
- The KYC status model must support DRAFT, INCOMPLETE, READY_TO_SUBMIT, SUBMITTED, UNDER_REVIEW, RESUBMISSION_REQUIRED, REJECTED, VERIFIED, and ACTIVE.
Account restriction logic:
- Incomplete users must not access bookings, payments, operations, dispatch, live tracking controls, finance modules, fleet activation, marketplace actions, or warehouse operations.
- Limited profile and verification pages may remain accessible so the user can complete KYC and view review status.
- Backend APIs must enforce restrictions server-side; frontend redirects are a convenience layer only.
Pending approval page rule:
- Pending approval may only appear after all role-required documents have been uploaded and the KYC packet has been submitted successfully.
- Users who have not submitted all required KYC must never be sent directly to Pending Approval.
Role-based admin review flow:
- Admin verification controls must filter by role, KYC state, document type, rejection reason, resubmission state, expiry risk, and submission date.
- Admins must review KYC by role type and approve, reject, request resubmission, add remarks, and view audit history.
- Review decisions must be document-level first, then account-level status must be recalculated from the role-required document set.
KYC lifecycle management:
- Users may save KYC drafts before final submission.
- Upload previews must be shown before submission where safe.
- Rejected/resubmitted documents must preserve history and create a new review cycle.
- The KYC wizard must be mobile responsive and support camera-first uploads for live-photo or identity evidence.
Security requirements:
- KYC uploads must use private secure storage, signed access URLs, MIME validation, file-signature validation, file size restrictions, upload rate limiting, encrypted sensitive metadata, and immutable audit trails.

63. Full Warehouse Management & Booking System
Objective
The warehouse module is not just storage. It is a full operational system covering customer warehouse booking, inbound appointment scheduling, receiving, inventory control, picking, packing, dispatch, cold chain, value-added services, reporting, and billing.
63.1 Warehouse Types & Storage Modes
Supported warehouse modes:
- Ambient / Dry Store: standard temperature storage for FMCG, general cargo, and electronics.
- Cold Store (2-8 C): refrigerated storage for pharmaceuticals, perishables, and dairy.
- Frozen Store (-18 C): deep-freeze storage for frozen food and sensitive temperature-controlled goods.
- Bulk / Floor Storage: open floor allocation for pallets, heavy goods, and oversized items.
- Bonded Warehouse: customs-controlled inventory for import/export handling.
- Open Yard: secured outdoor area for containers, vehicles, and oversized cargo.
Every warehouse location must declare:
- storage modes supported
- zone-level capacity
- operating hours
- service radius
- dock count and dock type
- special handling capability (hazmat, cold chain, bulky, high-value)
Warehouse physical hierarchy must support:
- Warehouse -> Zone -> Rack -> Row or Shelf -> Bin
- every bin must be uniquely addressable and queryable in real time
Minimum bin master data:
- bin_id
- zone_id
- rack_id
- row_or_shelf_ref
- storage_type
- max_weight_kg
- max_volume_cbm
- occupancy_status
- current_owner_id
- current_sku
- current_quantity
Additional structure rules:
- zone types may include Dry, Cold, Bonded, Open Yard, and Quarantine with separate pricing where commercial rules require it
- warehouse operations must expose a visual rack/bin map for occupancy visibility
- one owner per bin remains the default enforcement rule; any shared-bin exception requires explicit product and contract enablement plus auditable segregation controls
63.2 Customer Warehouse Booking Portal
Customers, Corporates, and Courier Companies must be able to self-book warehouse space from the app or portal without a manual phone workflow.
Booking flow:
- user selects Warehouse and Storage from service menu
- system shows eligible warehouses on map with live availability
- user selects storage type, expected quantity or volume, weight, inbound date and time, expected storage duration, and handling needs
- system checks real-time capacity and available inbound slot availability
- booking is confirmed with a Warehouse Booking Reference (WBR)
- system issues inbound appointment slot, access instructions, and QR or digital reference for gate check-in
Minimum warehouse booking fields:
- storage type
- goods description
- quantity and/or volume
- weight
- inbound date and time
- expected storage duration
- special handling requirements
- insurance required flag
- associated transport booking, LR, or waybill where applicable
Warehouse discovery and commercial rules:
- warehouse search must support filters for location, storage type, available capacity, price, and partner rating
- booking quote must be visible before confirmation
- booking confirmation may depend on advance payment, credit approval, or contracted billing terms
63.3 Dock & Inbound Appointment Scheduling
Each warehouse must support structured dock-slot scheduling.
Rules:
- each warehouse defines number of docks, dock type, operating hours, and slot capacity
- inbound and outbound slots are exclusive and cannot be overbooked
- slot availability is visible in real time to customer and warehouse operations
- arriving vehicles trigger check-in, driver verification, dock assignment, and GRN initiation
- outbound dispatch must support separate appointment scheduling
- no-show slot grace period is configurable; default policy is 30 minutes before release
Inbound appointment fields must support:
- date
- time window
- vehicle type
- pallet or unit count
- driver contact
Outbound release requests must support:
- requested SKU list
- quantities
- delivery address
- preferred dispatch date
Partner actions:
- warehouse partner may accept, reschedule, or reject an appointment with reason
- configurable no-appointment policy may reject unannounced vehicles
63.4 Goods Receiving Note (GRN) System
Every inbound warehouse movement must generate a GRN.
GRN lifecycle:
initiated -> in_progress -> completed -> discrepancy_raised -> closed
GRN process:
- vehicle checks in at gate or dock
- warehouse operator starts receiving workflow
- all cartons, pallets, parcels, or units are scanned during offload
- received quantity is compared to expected manifest or booking
- damage, shortage, overage, and mismatch are flagged immediately
- receiving operator captures evidence and signs off digitally
- accepted goods are allocated to storage location or cross-dock flow
Minimum GRN fields:
- GRN ID
- WBR / LR / Waybill reference
- goods owner / customer
- expected quantity
- received quantity
- condition on receipt
- receiving operator ID
- receiving timestamp
- discrepancy notes
- photo proof where required
- assigned storage bin or dispatch destination
Discrepancies must automatically raise:
- loss detection event
- warehouse issue / support event
- claim workflow input where damage or shortage is present
GRN completion controls:
- GRN PDF must include GRN number, warehouse, customer, SKU list, quantities, condition notes, receiver signature, and timestamp
- signed GRNs must become audit-locked after completion
- short, excess, and damaged receipt variance must be preserved as separate discrepancy categories
63.4.1 Damage At GRN And Quarantine Control
When damage is detected during GRN:
- receiver must classify damage type, capture photo evidence, and record affected quantity
- affected inventory must move to a Quarantine Zone state until resolved
- customer must be notified immediately with report and evidence
- customer options must include accept with note, raise claim, or request return / reverse movement
- quarantine overrides require mandatory admin reason logging
- quarantine aging must be reported, with items older than 14 days flagged for escalation
Damage claim SLA baseline:
- initial response target: 48 hours
- full resolution target: 7 days unless contract overrides apply
63.5 SKU & Product Catalog Management
Repeated warehouse clients must be able to maintain a SKU catalog.
Capabilities:
- customer creates SKU code, product name, category, UOM, dimensions, weight, storage rule, and expiry behavior
- inbound GRN may reference SKU by barcode or manual match
- stock levels are tracked per SKU by warehouse, zone, rack, and bin
- reserved, available, on-hand, damaged, and in-transit quantities are separated
- minimum stock thresholds generate alerts
- ownership visibility is limited to the owning client and authorized warehouse staff
Default SKU data must also support:
- barcode / QR / EAN reference
- fragile, hazmat, temperature-sensitive, and expiry-tracked flags
- bulk CSV import for large catalogs
- duplicate barcode detection within the same owner scope
- full SKU movement history from GRN through dispatch
63.6 Bin Location & Storage Assignment
Storage hierarchy:
Warehouse -> Zone -> Rack -> Shelf -> Bin
Rules:
- every bin stores capacity by unit, weight, or volume
- temperature and storage constraints are declared at zone/bin level
- bin suggestion may be automatic based on capacity, FIFO, FEFO, and storage rules
- manual override requires reason logging
- multi-client inventory is allowed only with ownership separation enforced
- no untracked storage movement is allowed; every movement must be scanned and auditable
Put-away execution rules:
- staff must scan both bin and item to confirm put-away
- system must block put-away when capacity is exceeded, zone rules are violated, or the bin is already reserved for another owner
- fast-moving SKUs may be preferentially placed near dispatch areas when configured
Shared-bin prevention must also be enforced at data and workflow level:
- one active owner_id per bin is the default database rule
- booking allocation and put-away validation must reject assignment into a bin already occupied by another owner unless a future shared-storage product mode explicitly overrides it
63.7 Pick, Pack & Dispatch Operations
Outbound warehouse operations must support:
- dispatch order creation from customer portal or internal operations
- pick-list generation by SKU, quantity, and optimal route through the warehouse
- assignment of pick task to operator or work queue
- mandatory scan at pick, pack, and vehicle load stages
- packaging and relabelling workflow where needed
- outbound manifest generation
- Goods Dispatch Note (GDN) creation
- customer notification with dispatch confirmation and tracking link
Rules:
- wrong-SKU scans must be blocked or forced into exception handling
- stock must reduce immediately after dispatch confirmation
- every outbound movement must remain traceable to booking, owner, operator, vehicle, and timestamp
Pick / pack execution details:
- pick list must show SKU, quantity, bin location, FIFO / FEFO order, and priority
- partial pick shortages must alert customer and supervisor before dispatch
- packing workflow must record packaging type, final weight, and dimensions where applicable
- dispatch note must bind picked SKU list, quantities, vehicle, and driver assignment
- optional secondary scan quality check may be enabled before sealing
63.8 Cross-Docking Workflow
Cross-docking must be a first-class warehouse operation.
Rules:
- booking or ops team may flag cargo for cross-dock
- goods are received, scanned, sorted, and moved to outbound dispatch without entering long-term storage
- inbound reference and outbound reference remain linked
- scan chain must preserve full audit trail
- same-day cross-dock avoids storage billing and applies handling charges only
63.8.1 Warehouse-To-Warehouse Transfer
Inter-warehouse transfer must be a supported inventory operation.
Rules:
- transfer request must capture source warehouse, destination warehouse, owner, SKU or batch list, quantity, transport reference, and requested transfer date
- source inventory must move into a reserved transfer state before dispatch
- outbound scan chain from source warehouse must link to inbound GRN or receipt workflow at destination warehouse

66. App Design System & UX Specification
Objective
The product experience must be defined as rigorously as the operations engine. Zito must feel like a modern logistics product, not a data-entry portal. The UI model is map-first, card-native, action-forward, and mobile-credible, with standards strong enough to outperform industrial benchmarks such as Porter-style booking flows while remaining Africa-ready and offline-aware.
66.1 The Core Design Problem - Forms vs Cards vs Map
The booking experience must not default to long forms. The map is the interface; cards and bottom sheets are the primary interaction surfaces.
- The first customer booking interaction must be route intent, not operational detail. The first 2 actions must read like:
  - Where from?
  - Where to?
- Pickup and Drop-off must behave like modern consumer apps: a search-led location selector with map fallback, not a technical split panel with too many early fields.
- The customer must be able to search and select a place for both pickup and drop-off before using move-pin refinement.
- Once either stop is active, the map must be visually dominant; small boxed map widgets are insufficient for the primary route-capture experience.
- Contacts, notes, pricing controls, and other secondary fields must wait until after route confidence is established.
Required design shifts:
- pickup and drop are map pin or search-overlay actions, not plain text form blocks on the first screen
- vehicle selection is visual card-based with live pricing, not a dropdown
- customer-visible pricing must show quote outcome, not internal pricing scope logic; county/town/rural classification and pricing-engine decisions remain hidden from the customer surface
- customer booking must support 2 customer-safe pricing outcomes:
  - instant quote available: show the customer-safe quote before confirmation
  - instant quote unavailable: show a neutral rate-under-review state and still allow booking request submission for admin, contract, or marketplace follow-up
- raw pricing-engine failures such as "no active rate card" are forbidden on customer surfaces; internal rate, county, contract, and engine diagnostics belong only to admin/internal workflows
- where instant pricing is unavailable, admin must be able to confirm the final rate from contract terms, negotiated pricing, or marketplace proposals from transporters, drivers, and partners before customer confirmation is finalized
- when map/geocode data is available, pricing scope must default automatically from the selected pickup and route context; any override belongs to admin/internal desks only
- cargo details are progressive and mostly optional unless operationally mandatory
- booking confirmation is a concise summary card or bottom sheet, not a separate paperwork-style page
- booking history is card feed first; table views are secondary and report-oriented
66.2 Design Language - The Zito System
Core design principles:
- Map-first: every location-heavy workflow begins from the map
- Card-native: content appears in cards, tiles, chips, sheets, and status trails rather than grids of raw data
- Action-forward: the primary action is always visually obvious
- Progressive disclosure: ask only for what is needed now
- One-thumb design: primary mobile controls remain in the lower interaction zone
- Offline-tolerant: every critical screen has a meaningful degraded state

66.17 Product Hardening Before External Go-Live
Before company-registration-dependent integrations are finalized, the product must still be operationally strong, deployment-ready, and internally testable.
Deployment readiness rules:
- the current web delivery assumption is split frontend and backend deployment on Vercel; frontend and mobile runtime configuration must not retain obsolete Railway fallback hosts once Vercel deployment is active
- deployed frontend environments must expose an explicit backend origin when frontend and backend run as separate Vercel projects
- mobile runtime defaults may use the deployed Vercel backend origin for QA convenience, while local development may override that host explicitly through environment files
Freight and container operational depth rules:
- rail and container bookings must expose a visible milestone chain rather than hiding all freight progress inside a single booking status
- milestone coverage must include at minimum document pack readiness, customs clearance, PAC readiness, origin handoff, linehaul, ICD or destination gate handoff, last-mile dispatch, and delivery confirmation where relevant
- admin freight control desks must allow milestone status updates, notes, and blocked-state capture without leaving the booking control workflow
- customer-facing freight bookings must expose a readable milestone timeline so import/export and corridor customers can understand where the shipment is delayed or progressing
Autobot v2 support rules:
- support ticket creation must persist autobot confidence, suggested quick action, escalation desk, and suggested first-response guidance in addition to the matched article summary
- customer support creation should show the suggested desk and suggested reply posture before ticket submission so the workflow feels guided rather than form-driven
- internal support queues and ticket threads must surface autobot routing metadata clearly enough for customer care, operations, accounts, or compliance desks to take action without re-triaging from scratch
Compliance automation rules:
- verification dashboards must surface expiring approved documents, expired documents, overdue user reviews, overdue vehicle reviews, missing photo packs, and any users auto-suspended by compliance automation
- operational roles affected by expired mandatory compliance evidence may be auto-suspended according to policy, but those actions must be visible in the verification control desk
- verification control must support explicit automation refresh so compliance teams can rerun the latest scan before taking manual decisions
Admin workflow polish rules:
- the main admin dashboard must surface cross-desk priorities, not just isolated module links
- compliance backlog, automation alerts, escalated support pressure, and finance-routed support pressure must all be visible from the main command surface
- quick actions must include direct access to the verification desk and freight-control desk
Protected workspace session rules:
- admin, internal, partner, agency, and other protected workspaces must require a valid authenticated user and a valid access token together; stale user-only session remnants must not be treated as authorized access
- a `401 Unauthorized` response from a protected workspace API call must clear the broken session and redirect the operator back to the correct portal login surface instead of leaving the user on a half-rendered control page with only an inline unauthorized error
Mobile QA hardening rules:
- the internal mobile workspace must expose a QA diagnostics screen showing the active API host, runtime environment label, token presence, user-role routing context, and live backend health response
- mobile API clients must use a bounded request timeout and return actionable errors when the runtime cannot reach the deployed backend
- mobile documentation and environment templates must reflect the active deployment strategy and must not reference obsolete infrastructure defaults
- installed Android and iPhone builds must resolve the root app launch route correctly; opening the app from the launcher or root deep link scheme must never show an unmatched-route screen
- the root mobile launch path must always land on a valid auth entry or role-routed home flow
Registration and account-provisioning UX rules:
- any registration, account-creation, staff-provisioning, partner-provisioning, or driver-onboarding form that captures a phone number must capture country code and local number as separate visible inputs; hidden or assumed country code behavior is not acceptable
- phone entry defaults may be preselected per market, but the country code must remain user-editable in every registration and provisioning flow
- create-user, create-staff, create-partner-account, and create-driver actions must show inline success or failure feedback at the same point as the submit action, not only in distant page-level alerts
- successful provisioning flows must offer an immediate "Create another" continuation action so back-office teams can onboard multiple users without manual form reconstruction
